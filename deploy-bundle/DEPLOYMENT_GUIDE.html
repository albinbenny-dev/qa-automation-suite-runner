Build Images on Dev Machine

2
Prepare Deploy Bundle
Dev Machine — PowerShell
Create folder structure
New-Item -ItemType Directory -Force deploy-bundle\images
New-Item -ItemType Directory -Force deploy-bundle\nginx
Save all Docker images to tar files
docker save qaasr-api:latest    -o deploy-bundle\images\qaasr-api.tar
docker save qaasr-ui:latest     -o deploy-bundle\images\qaasr-ui.tar
docker save qaasr-runner:latest -o deploy-bundle\images\qaasr-runner.tar
docker save postgres:16-alpine  -o deploy-bundle\images\qaasr-postgres.tar
docker save redis:7-alpine      -o deploy-bundle\images\qaasr-redis.tar
postgres and redis only needed for a fresh server install — skip them for update deployments (use .\deploy.ps1 instead).
Copy config files
Copy-Item docker-compose.yml  deploy-bundle\
Copy-Item .env               deploy-bundle\
Copy-Item nginx\nginx.conf   deploy-bundle\nginx\
Export Postgres database — write dump file from inside the container (avoids Windows UTF-16 encoding corruption from PowerShell > redirect)
docker exec qaasr-postgres pg_dump -U qasr qasr -f /tmp/qasr-dump.sql
docker cp qaasr-postgres:/tmp/qasr-dump.sql deploy-bundle\qasr-dump.sql
Export robot scripts volume
docker run --rm `
  -v qa-automation-suite-runner_qaasr-scripts:/scripts `
  -v "${PWD}/deploy-bundle:/backup" `
  alpine tar czf /backup/qaasr-scripts.tar.gz -C /scripts .
ℹ
Artifacts (videos/screenshots) and run history are intentionally excluded — fresh start on the server.

Expected bundle structure

deploy-bundle/ ├── images/ │ ├── qaasr-api.tar │ ├── qaasr-ui.tar │ ├── qaasr-runner.tar │ ├── qaasr-postgres.tar │ └── qaasr-redis.tar ├── nginx/ │ └── nginx.conf ├── docker-compose.yml ├── .env ├── qasr-dump.sql ← database export └── qaasr-scripts.tar.gz ← robot script files

6
Start the Stack
Linux Server — Bash
⚠
Run from the parent folder (/data/autoab/qa-automation-suite-runner), not from inside deploy-bundle/. Docker Compose looks for .env and docker-compose.yml in the current directory — if they are only inside deploy-bundle/ the POSTGRES_PASSWORD variable will be missing and startup will fail.
Copy config files to the parent folder first
cp deploy-bundle/.env .
cp deploy-bundle/docker-compose.yml .
Start all services from the parent folder
sudo docker-compose -p qa-automation-suite-runner up -d
Check all containers are running
sudo docker-compose -p qa-automation-suite-runner ps
Check API logs for startup errors
sudo docker logs qaasr-api --tail 20
7
Restore Data
Linux Server — Bash
⚠
Wait ~15 seconds after up -d for Postgres to fully initialize before restoring.
Restore Postgres database
cat qasr-dump.sql | sudo docker exec -i qaasr-postgres psql -U qasr qasr
Restore robot scripts volume
sudo docker run --rm \
  -v qa-automation-suite-runner_qaasr-scripts:/scripts \
  -v $(pwd):/backup \
  alpine tar xzf /backup/qaasr-scripts.tar.gz -C /scripts
✕
The API runs prisma db push on startup and creates the schema automatically. If you restore the dump into a database that already has tables you will get relation already exists errors. Always stop the API first, drop the database, recreate it, then restore.
Stop API and runner first — they hold active DB connections that block the drop
sudo docker stop qaasr-api qaasr-runner
Drop and recreate the database cleanly
sudo docker exec -i qaasr-postgres psql -U qasr postgres -c "DROP DATABASE qasr;"
sudo docker exec -i qaasr-postgres psql -U qasr postgres -c "CREATE DATABASE qasr;"
Restore the dump into the clean database
cat qasr-dump.sql | sudo docker exec -i qaasr-postgres psql -U qasr qasr
Start API and runner back up
sudo docker start qaasr-api qaasr-runner
Verify data restored correctly
sudo docker exec -i qaasr-postgres psql -U qasr qasr -c "SELECT email FROM \"User\";"

Future Updates — Use deploy.ps1
Dev Machine — PowerShell
ℹ
deploy.ps1 at the repo root automates all update steps. Run it from the project folder in PowerShell. It builds images, saves tars, SCPs to the server, loads, and restarts — with a health check and disk usage summary at the end.
Code changes only — build, transfer, rolling restart
.\deploy.ps1
Full migration — code + DB dump + scripts volume restore
.\deploy.ps1 -Mode full
Prompts "Type YES" before touching the database. Stops api+runner, drops/recreates DB, restores dump and scripts volume, then starts everything back up.
Override SSH alias or remote path if needed
.\deploy.ps1 -SSH my-server -RemoteDir /opt/qasr
✕
Never run docker-compose down -v — the -v flag deletes all volumes including the database.

Troubleshooting

!
DB restore fails with "relation already exists"
DB Error
⚠
The API runs prisma db push on startup and pre-creates the schema. Restoring the dump into an already-initialised database causes conflicts. Stop the API first to release DB connections, then drop and recreate cleanly.
Stop containers that hold DB connections
sudo docker stop qaasr-api qaasr-runner
Drop and recreate the database
sudo docker exec -i qaasr-postgres psql -U qasr postgres -c "DROP DATABASE qasr;"
sudo docker exec -i qaasr-postgres psql -U qasr postgres -c "CREATE DATABASE qasr;"
Restore the dump then start containers
cat deploy-bundle/qasr-dump.sql | sudo docker exec -i qaasr-postgres psql -U qasr qasr
sudo docker start qaasr-api qaasr-runner
!
Video recording fails — "Cannot open display :99"
Runner Error
⚠
Xvfb (virtual display) failed to start during container startup. The set -e in the entrypoint caused x11vnc to exit early when the display wasn't ready yet, leaving Xvfb dead.
Check if Xvfb, openbox and x11vnc are running
sudo docker exec qaasr-runner ps aux | grep -E "Xvfb|x11vnc|openbox"
All three should appear. If any are missing, proceed below.
Restart the runner container — the entrypoint will re-run and start Xvfb cleanly
sudo docker restart qaasr-runner
sleep 5
sudo docker exec qaasr-runner ps aux | grep -E "Xvfb|x11vnc|openbox"
Expected output after successful restart — all three processes visible
root   Xvfb :99 -screen 0 1280x900x24 -ac +extension GLX +render -noreset
root   openbox
root   x11vnc -display :99 -forever -nopw -quiet -rfbport 5900
!
POSTGRES_PASSWORD missing — stack fails to start
Config Error
⚠
Running docker-compose up from inside deploy-bundle/ means the .env file is not found in the current directory.
Copy config files to the parent folder and run from there
cp deploy-bundle/.env .
cp deploy-bundle/docker-compose.yml .
sudo docker-compose -p qa-automation-suite-runner up -d

Server-Specific Configuration

H
Internal Hostname Resolution (extra_hosts)
docker-compose.yml
ℹ
If the target application uses an internal hostname (e.g. *.ocplab.6d.local) that is not in public DNS, Chrome inside the runner container cannot resolve it. Add extra_hosts to map the hostname to its IP permanently.
Add extra_hosts under qaasr-runner in docker-compose.yml
  qaasr-runner:
    image: qaasr-runner:latest
    container_name: qaasr-runner
    restart: unless-stopped
    extra_hosts:
      - "airtel6d-in-ventas-master-int-aavm-alpha-01.ocplab.6d.local:10.0.12.244"
    environment:
Recreate the runner container to apply
sudo docker-compose -p qa-automation-suite-runner up -d --no-build qaasr-runner
Verify hostname resolves inside the container
sudo docker exec qaasr-runner cat /etc/hosts | grep ocplab
⚠
Do NOT add the bind mount to the top-level volumes: block — extra_hosts goes under the service definition only. Top-level volumes are only for named Docker volumes.
C
Trusting Internal CA Certificates (SSL)
Dockerfile + Rebuild
ℹ
If the target app uses a self-signed or internal CA certificate, Chrome will reject the connection. Bake the cert into the runner image so it is always trusted — no manual injection needed after restarts.

Permanent fix — bake into image (recommended)

On dev machine — pull the cert from the server and place it in the certs folder
scp qa-server:/data/autoab/ocplab-ca.crt packages\runner\certs\ocplab-ca.crt
Any .crt file in packages/runner/certs/ is automatically trusted during image build.
Rebuild and transfer the runner image
docker compose -p qa-automation-suite-runner build qaasr-runner
docker save qaasr-runner:latest -o qaasr-runner-update.tar
scp qaasr-runner-update.tar qa-server:/data/autoab/qa-automation-suite-runner/
On server — load and recreate
sudo docker load -i qaasr-runner-update.tar
sudo docker-compose -p qa-automation-suite-runner up -d --no-build qaasr-runner
Verify SSL connection works
sudo docker exec qaasr-runner curl -v \
  "https://airtel6d-in-ventas-master-int-aavm-alpha-01.ocplab.6d.local/" \
  2>&1 | grep -E "SSL|certificate|Connected"

Temporary fix — inject into running container (lost on restart)

sudo docker cp /data/autoab/ocplab-ca.crt \
  qaasr-runner:/usr/local/share/ca-certificates/ocplab-ca.crt
sudo docker exec qaasr-runner update-ca-certificates
